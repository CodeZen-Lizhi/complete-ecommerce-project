# 全阶段编号练习技术设计

## 1. 边界与目录契约

- 根目录固定为 `examples/ai/phase1/` 至 `examples/ai/phase7/`。
- 阶段内目录采用两位数字序号和英文 snake_case：`NN_<slug>`。
- 序号完全跟随路线图“练习顺序”，不跨阶段连续计数。
- 每个目录至少包含 `README.md`；需要 Go 编码的练习增加 `main.go`，阶段 3–7 使用 `exercise.go` 保存题目专属契约，适合测试驱动的练习增加最小 `_test.go` 骨架。
- 阶段 1 是导航型练习：README 指向现有项目层级和待实现位置，不复制生产代码。
- 阶段 2 至阶段 7 是隔离型学习示例：不导入生产 Handler、Container 或运行时单例，只有明确需要复用电商领域接口的 Tool/Agent 练习才通过小接口表达边界。

## 2. 练习骨架契约

每个 README 使用统一结构：

1. 练习目标
2. 前置知识
3. TODO 顺序
4. 运行或验证方式
5. 完成标准
6. 暂不实现

每个 Go 骨架遵循以下约束：

- 所有新增函数和方法写简洁中文职责注释。
- 核心步骤使用 `TODO 1`、`TODO 2` 连续编号，按真实执行顺序排列。
- 默认配置先做占位符校验，再创建 Redis、数据库、模型或向量库客户端。
- 未完成实现通过统一的包内错误值或明确错误文本退出，不伪造成功结果。
- 外部组件通过小接口或配置结构隔离，避免为了让骨架编译而引入真实连接。
- 练习目录之间不互相导入，避免前一题答案成为后一题的隐式依赖。

## 4. 分阶段任务树

- 阶段 1：7 个电商项目导航型练习。
- 阶段 2：迁移前 6 个练习并生成第 7 至第 11 个骨架。
- 阶段 3：12 个 RAG 骨架。
- 阶段 4：7 个 Tool Calling 骨架。
- 阶段 5：8 个 Agent 工作流骨架。
- 阶段 6：7 个 Eval 骨架。
- 阶段 7：8 个安全、监控与部署骨架。
- 父任务：维护全量练习索引、路线文档引用、全仓验证和最终一致性 review。

## 5. 依赖与文档策略

- 优先复用 `go.mod` 中已有依赖；只有某一阶段骨架必须引用真实公开类型时才增加对应 Eino 扩展依赖。
- 具体 Eino API 在实现相应阶段前查当前官方文档，不按记忆猜测签名。
- 需要 Docker、MySQL、Redis、向量库或模型服务的练习，README 提供环境变量和启动前置条件；默认骨架不自动启动或连接外部服务。
- 路线文档增加编号目录索引，README 中的运行命令统一指向新路径。

## 6. 兼容性与回滚

- 目录重命名会改变用户已有运行命令，这是有意的可见变化；当前文档和命令中的旧路径必须同步替换，归档任务保留历史快照。
- 每个阶段作为独立提交单元，出现问题可按阶段回滚，不影响其他阶段。
- 不删除已有练习实现；迁移验证通过前保留内容级对比证据。
- 若新依赖导致全仓构建不稳定，优先将骨架改为本地接口而不是保留无必要依赖。
